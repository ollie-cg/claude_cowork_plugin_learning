# Plugin Brands Weekly Client Update Deck — v1

One deck = one brand. No cross-brand comparisons. No cross-deal comparisons.

## Inputs (both required)

1. **Pipeline MD file** — `docs/temporary/{brand-slug}-pipeline-{YYYY-MM-DD}.md`, generated by the `brand-pipeline-report` skill. Source of deal-level detail.
2. **Weekly pipeline spreadsheet** — the brand's row. Source of topline stage counts that the client sees elsewhere. Numbers must match the deck.

If the spreadsheet row and the MD summary table disagree, prefer the spreadsheet (it's what the client has) and flag the gap in the rep-facing note.

## Personalisation questions (ask up to 3; skip if provided)

1. Rep name (defaults to the most common Owner in the MD's in-play deals).
2. Rep headshot for cover? (optional)
3. Execution images for appendix? (optional)

Client website / logo direction can be inferred from the brand's public brand guidelines; don't block on it.

## Procedure

### 1. Parse the MD

- Read the `## Summary` table → stage counts + total unique deals.
- Read each stage section (`## {Stage} (N)`) → list of deal blocks with buyer name, owner, sites, route to market, last contact date, summary, recent exchanges.

### 2. Pick the Top 5 customers

**In-play only.** Exclude Won, Lost, No Response (Won is spotlighted on slide 5; Lost and No Response do not appear in this deck).

**Rank by stage seniority, then by recency within stage.** Stage order (most advanced first):

1. Proposal Feedback Pending
2. Proposal
3. Feedback Received
4. Feedback Pending
5. Follow Up
6. Discovery

Within each stage, sort by `last contact` date descending (most recently touched first). Deals with no recorded contact go last. Take the first 5 across the combined ranking.

If fewer than 5 in-play deals exist across those six stages, take all of them; don't pad with Won/Lost.

### 3. Collect Won deals

All deals in the `## Won (N)` section are listed on slide 5 with buyer name, sites, and route to market. No ranking. No "biggest / best" language.

### 4. Build the Gamma markdown (preserve mode)

Produce one markdown block with exactly 5 sections, separated by `---`. Gamma's `textMode: "preserve"` keeps each section as one card.

Section order — **fixed**:

1. Cover
2. Scope / Remit
3. Pipeline Summary
4. Top 5 Customers in the Buyer Pipeline
5. Wins & Next Focus

### 5. Slide rules

**Slide 1 — Cover**
- Brand name (verbatim from MD header), reporting period, rep name.
- Plugin Brands wordmark (with the full stop).
- Black background, gold/amber `#FDBC58` accent, white supporting.

**Slide 2 — Scope / Remit** (What / So What / Now What)
- Describe commercial remit for this brand.
- Conservative: don't invent KPIs. Don't mention other clients.

**Slide 3 — Pipeline Summary** (What / So What / Now What)
- Stage counts from the spreadsheet row. Total unique deals.
- So What: proportion in advanced vs prospecting stages; overall shape.
- Now What: where the conversion pressure is.
- **Do NOT use comparative language** about other brands or deals.

**Slide 4 — Top 5 Customers in the Buyer Pipeline**
- Heading: "Top 5 Customers in the Buyer Pipeline" + subtitle "Most advanced in-play deals".
- One block per deal, in the ranked order from step 2. Each block self-contained.
- **Per-deal fields:** buyer name, stage, sites (if known), route to market (if known), owner, one-line state. **Do not include a "last contact" date** — it's internal CRM hygiene, not client-facing signal.
- **No cross-deal language.** No "ahead of", "behind", "stronger than", "the standout", "the leader", "outperforms", "vs".
- **No ranking language.** No "#1", "top performer", "best", "biggest". Numbering 1–5 is fine (it's just order).
- Drop the What / So What / Now What frame on this slide — per-deal detail doesn't fit cleanly.

**Slide 5 — Wins & Next Focus** (What / So What / Now What)
- Wins secured: every Won deal named with sites + route to market.
- Next focus: bullets derived from the MD — which stages are clogged, which specific in-play deals need reviving, where the next week's attention goes.
- **No "this win was bigger than X" comparisons.**

**Appendix** (optional — only if execution images uploaded)

### 6. POST to Gamma

```
POST https://public-api.gamma.app/v1.0/generations
Headers: X-API-KEY: $GAMMA_API_KEY  (from project .env)
Body:
{
  "inputText": "<the 5-section markdown>",
  "textMode": "preserve",
  "format": "presentation",
  "numCards": 5,
  "additionalInstructions": "Black background. Gold/amber accent #FDBC58. White supporting. Plugin Brands wordmark with full stop. Minimal, bold, commercial. Client update tone — external-facing.",
  "cardOptions": { "dimensions": "16x9" }
}
```

Poll `GET /v1.0/generations/{id}` every 5s until `status: completed` (typically ~25–30s). Return `gammaUrl`.

## Data rules — hard constraints

- Use only the MD, the spreadsheet row, and user-provided inputs.
- If a buyer's summary says "no {BRAND}-specific mention appears" — describe the deal by its stage mechanics, not by inventing buyer intent for this brand.
- Never mention other brands in the Plugin Brands portfolio on a client-specific deck.
- Never compare one deal to another.
- Zero counts that are genuinely zero: state plainly (e.g. "Feedback Received: 0"), don't emphasise.
- If the most recent engagement on a top-5 deal is an auto-reply / OOO, say so — don't dress it up.

## Output

Return to the user:
1. The Gamma share URL (`gammaUrl`).
2. The 5-section markdown used as input (saved to the project folder for re-runs).
3. A one-line note of which fields were defaulted (e.g. "Rep defaulted to Danny Armstrong — 5 of 5 in-play top deals owned by him").

Never output the raw Gamma prompt without also running the generation — v0's "paste-it-yourself" flow is replaced by direct API call.
